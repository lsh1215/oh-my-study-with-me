# Deep Study - First Principles 학습 스킬

## 인자 파싱
- 인자 없음 → 책 선택부터 시작
- `[책이름 키워드]` → 해당 책 바로 시작
- `continue` → 이전 학습 이어서 진행 (진행 파일 참조)

인자: $ARGUMENTS

## Phase 0: 책 선택

1. `책 목록/` 하위 폴더와 PDF 파일 목록을 보여준다.
2. 사용자가 책을 선택하면, PDF 앞부분(1~10페이지)을 읽어 **목차를 추출**한다.
3. 목차를 보여주고, 어떤 챕터부터 시작할지 사용자에게 묻는다.

## Phase 1: 챕터 읽기 + 핵심 원리 추출

1. 선택된 챕터의 페이지 범위를 PDF에서 읽는다 (20페이지씩).
2. 읽은 내용에서 **핵심 원리**를 추출한다.
   - 단순 요약 금지. "이 챕터가 풀려는 근본 문제는 무엇인가?"를 중심으로 정리.
   - 형식:
     ```
     ## [챕터명]

     ### 근본 문제
     이 챕터가 다루는 핵심 문제가 무엇인지

     ### 핵심 원리
     - 원리 1: [왜 이런 방식인지까지 포함]
     - 원리 2: ...

     ### 연결고리
     - 다른 개념/기술과의 관계
     ```
3. 추출한 원리를 사용자에게 보여주고, Phase 2로 넘어간다.

## Phase 2: First Principles 대화

사용자와 "왜?" 를 끝까지 파고드는 대화를 진행한다.

### 대화 가이드라인
- 사용자가 설명하면, 항상 한 단계 더 깊은 "왜?"를 물어본다.
- 사용자의 답변이 틀리거나 불완전하면, 바로 정답을 주지 말고 힌트를 준다.
- 더 이상 파고들 수 없는 근본 원리에 도달하면 대화를 마무리한다.
- 대화 중 사용자가 모르는 부분이 나오면, PDF 원본 해당 부분을 다시 읽어서 설명한다.

### 대화 시작 방법
추출한 핵심 원리 중 하나를 골라 질문을 던진다:
- "이 기술이 왜 이런 설계를 선택했을까?"
- "만약 이 부분이 없다면 어떤 문제가 생길까?"
- "이것과 [유사 기술]의 근본적인 차이는 뭘까?"

사용자가 충분히 이해했다고 판단되면 Phase 3로 넘어간다.

## Phase 3: 유형별 검증

챕터의 콘텐츠 유형에 따라 적절한 검증을 수행한다.

### 유형 판별 기준
- **이론/개념** (분산 시스템 이론, CS 기초, 네트워크 등): 파인만 테스트
- **코드/구현** (자바, 코틀린, 스프링, JPA 등): 코드 구현 챌린지
- **설계/아키텍처** (시스템 설계, DDD, 마이크로서비스 등): 설계 문제
- **운영/트러블슈팅** (카프카 운영, DB 튜닝, JVM 튜닝 등): 장애 시나리오
- 하나의 챕터가 여러 유형에 해당하면, 가장 핵심적인 유형을 선택한다.

### 검증 방법

#### 파인만 테스트 (이론/개념)
- "이 개념을 비개발자(또는 주니어)에게 설명한다고 생각하고, 쉬운 비유를 들어서 설명해봐."
- 설명이 정확하고 쉬우면 통과. 아니면 어디가 부족한지 피드백.

#### 코드 구현 챌린지 (코드/구현)
- 이 챕터에서 배운 핵심 개념을 활용한 작은 코딩 과제를 낸다.
- 사용자가 구현하면, 코드 리뷰를 해주고 개선점을 제안한다.

#### 설계 문제 (설계/아키텍처)
- 실제 서비스 시나리오를 주고 설계하게 한다.
- 예: "하루 1000만 건의 주문을 처리하는 시스템을 설계해봐. 이 챕터에서 배운 원리를 적용해서."

#### 장애 시나리오 (운영/트러블슈팅)
- 실제 발생할 수 있는 장애 상황을 제시한다.
- 예: "레디스 메모리가 갑자기 90%를 넘었다. 모니터링 대시보드에서 이걸 발견했을 때, 어떤 순서로 대응할래?"

### 검증 결과 처리
- **통과**: "핵심을 정확히 이해하고 있다" + Phase 4로 진행
- **부분 통과**: 부족한 부분 피드백 → 해당 부분 PDF 재참조 → 재검증
- **미통과**: 핵심 원리 다시 설명 → Phase 2부터 재시작

### 퀴즈 뱅크 자동 저장 (Slack 일일 복습 연동)

Phase 3에서 사용한 검증 질문을 `quizzes/quiz_bank.json`에 자동 추가한다.
(`/oh-my-study-with-me:setup-quiz`로 퀴즈 시스템을 구축한 경우에만 동작)

추가할 퀴즈 형식:
```json
{
  "id": "{카테고리}-{순번}",
  "category": "카프카",
  "book": "카프카 핵심 가이드",
  "chapter": "Ch3. 프로듀서",
  "type": "concept|code|design|troubleshooting",
  "concept": "이 퀴즈가 검증하는 핵심 개념명",
  "question": "검증에 사용한 질문",
  "hint": "사고를 유도하는 힌트",
  "answer": "핵심 원리 기반 모범 답변",
  "keywords": ["핵심개념어1", "핵심개념어2"],
  "min_keyword_match": 2,
  "box": 1,
  "next_review": "내일 날짜",
  "times_asked": 0,
  "times_correct": 0,
  "last_asked": null,
  "slack_message_ts": null
}
```

규칙:
- keywords는 5~8개, 한글/영어 모두 포함 (예: ["GC", "가비지", "힙", "heap"])
- min_keyword_match는 키워드 수의 약 30% (반올림)
- 같은 question이 이미 존재하면 추가하지 않는다
- quiz_bank.json 파일이 없으면 이 단계를 건너뛴다
- concept는 tracking/{카테고리}.md의 개념 테이블에 있는 개념명과 일치해야 한다

### 개념별 메타인지 추적

Phase 3 검증 결과를 개념 단위로 추적한다. 챕터 단위의 "통과/미통과"를 넘어, 어떤 개념이 약하고 어떤 개념이 확실한지 정밀하게 기록한다.

#### 추적 파일 위치
`tracking/{카테고리}.md` (예: `tracking/카프카.md`, `tracking/엘라스틱서치.md`)

#### 파일 구조
```markdown
# [카테고리] 개념 추적

## 대시보드
| 상태 | 개수 |
|------|------|
| 🟦 체득 (90%+) | 0 |
| 🟩 양호 (70-89%) | 0 |
| 🟨 보통 (40-69%) | 0 |
| 🟥 약함 (0-39%) | 0 |
| ⬜ 미측정 | 0 |

## 개념별 현황

| 개념 | 출처 | 시도 | 정답 | 정답률 | 마지막 테스트 | 상태 |
|------|------|------|------|--------|-------------|------|
| acks 설정 | 카프카 핵심가이드 Ch3 | 3 | 3 | 100% | 2026-03-01 | 🟦 |
| 배치 전송 | 카프카 핵심가이드 Ch3 | 2 | 1 | 50% | 2026-02-28 | 🟨 |

## 에러 노트

### 배치 전송
- **혼동**: linger.ms와 batch.size의 관계를 반대로 이해
- **핵심**: linger.ms는 대기 시간, batch.size는 바이트 크기. linger.ms가 지나거나 batch.size에 도달하면 전송
- **날짜**: 2026-02-28
```

#### 업데이트 규칙

1. **Phase 3 검증 시**: 검증에서 다룬 각 개념을 개별 행으로 기록한다.
   - 검증 통과 → 해당 개념들 정답 +1
   - 부분 통과 → 맞힌 개념은 정답 +1, 못 맞힌 개념은 시도만 +1 + 에러 노트 작성
   - 미통과 → 모든 개념 시도만 +1 + 에러 노트 작성

2. **에러 노트 작성**: 사용자가 틀렸거나 불완전하게 답한 개념에 대해:
   - **혼동**: 사용자가 무엇과 혼동했는지
   - **핵심**: 올바른 이해
   - **날짜**: 에러 발생 날짜

3. **에러 노트 해소**: 이후 세션에서 해당 개념을 다시 맞히면 에러 노트에 `✅ 해소 (날짜)` 추가.

4. **상태 뱃지 기준**:
   - 🟦 체득: 정답률 90% 이상, 최소 3회 이상 시도
   - 🟩 양호: 정답률 70-89%
   - 🟨 보통: 정답률 40-69%
   - 🟥 약함: 정답률 0-39%
   - ⬜ 미측정: 시도 0회

5. **대시보드 갱신**: 개념 테이블이 업데이트될 때마다 상단 대시보드의 카운트도 갱신.

6. **파일이 없으면** tracking/ 디렉토리와 해당 카테고리 파일을 새로 생성한다.

#### Slack 복습 퀴즈 연동

quiz_bank.json의 퀴즈에도 `"concept"` 필드를 추가한다:
```json
{
  "concept": "acks 설정",
  ...
}
```
GitHub Actions 채점 시 이 concept 필드를 기준으로 tracking/{카테고리}.md도 함께 업데이트할 수 있도록 한다.

#### Phase 2 대화에서 활용

`continue`로 이전 학습을 이어갈 때, 해당 카테고리의 concepts 파일을 읽어서:
- 🟥 약함 상태인 개념이 있으면 Phase 2 대화에서 해당 개념을 우선 다룬다.
- 에러 노트가 있는 개념은 다른 맥락에서 재질문한다 (같은 질문 반복 금지).

## Phase 4: 학습 요약 메모

검증을 통과하면 이번 세션에서 배운 내용을 짧은 메모로 정리한다.

### 메모 내용
```markdown
# [주제명] - 학습 메모

- **근본 문제**: [이 챕터가 풀려는 핵심 문제 한 줄]
- **핵심 원리**: [Phase 2 대화에서 도달한 원리 2~3줄]
- **검증 결과**: [Phase 3에서 어떤 검증을 했고 결과는 어땠는지]
- **블로그 소재**: [글로 쓸만한 포인트가 있으면 메모. 나중에 `/oh-my-study-with-me:blog`로 작성]
```

### 저장
- `study-notes/{카테고리}/{책이름}/` 하위에 `{챕터명}.md`로 저장한다.
- 이 메모는 나중에 `/oh-my-study-with-me:blog`로 블로그를 쓸 때 소재로 활용된다.

> **참고**: 본격적인 블로그 글쓰기는 `/oh-my-study-with-me:blog` 스킬을 별도로 호출한다. 이 스킬은 학습에만 집중한다.

## Phase 5: 진행 기록

학습 진행 상황을 `books/{카테고리}/{책이름}_progress.md` 파일에 기록한다.

```markdown
# [책 이름] 학습 진행

## 완료한 챕터
- [x] Ch1. [챕터명] - 검증: 통과 - 메모: 작성완료
- [ ] Ch2. [챕터명]

## 개념 상태 요약
- 🟦 체득: N개 | 🟩 양호: N개 | 🟨 보통: N개 | 🟥 약함: N개
- 약점 개념: [개념1, 개념2, ...]
- 상세: `tracking/[카테고리].md` 참조

## 다음 학습 예정
- Ch2. [챕터명]

## 메모
- [학습 중 발견한 교차 참조 포인트 등]
```

## Phase 6: 다음 주제

1. 같은 책의 다음 챕터로 이어갈지, 관련된 다른 책의 챕터로 교차 참조할지 사용자에게 제안한다.
2. 교차 참조 제안 시, 현재 배운 개념과 연결되는 다른 책의 챕터를 구체적으로 알려준다.
   - 예: "카프카 프로듀서를 배웠으니, '이벤트 기반 마이크로서비스 구축'의 3장에서 이벤트 소싱 관점으로 보면 더 깊어질 수 있어."

## 주의사항

- 절대로 책 내용을 단순 복붙하지 않는다.
- 사용자가 "모르겠다"고 하면 바로 답을 주지 말고, 사고를 유도하는 질문을 먼저 한다.
- 검증 단계를 건너뛰지 않는다. 반드시 Phase 3을 거친 후 Phase 4로 넘어간다.
- PDF 읽기 시 한번에 최대 20페이지. 챕터가 길면 나눠서 읽는다.
